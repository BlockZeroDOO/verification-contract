## 3. Problem / Solution

### Problem

Большинство цифровых процессов страдают от одной и той же проблемы: данные есть, но доказать момент их существования, неизменность и последовательность действий сложно.

Когда спор доходит до проверки, компании сталкиваются с вопросами:

- когда именно был создан документ или набор данных
- изменялся ли он после публикации
- можно ли доказать его состояние в конкретный момент времени
- кто инициировал запись
- как быстро восстановить историю проверки

### Solution

deNotary решает эту задачу на уровне инфраструктуры.

Платформа:

- позволяет подготавливать данные к детерминированной фиксации локально в клиенте или через API
- создает криптографический hash
- публикует single или batch anchor
- отслеживает финализацию записи
- выдает receipt после finality
- предоставляет audit layer для поиска и проверки

В результате компания получает надежный цифровой механизм подтверждения, который легко встроить в существующие продукты и бизнес-процессы.

## 4. Product description

deNotary — это платформа цифровой верификации и доказуемой фиксации данных, без раскрытия самих данных.  
Она подходит для документов, сертификатов, отчетов, KYC/KYB-артефактов, журналов событий, результатов вычислений, AI-output, IoT-событий и любых цифровых записей, для которых важны неизменность и проверяемость.

Система поддерживает два базовых сценария:

- **Single anchoring** — когда нужно зафиксировать отдельную запись
- **Batch anchoring** — когда нужно эффективно подтверждать большие массивы данных через merkle root и manifest

Вокруг этого ядра построены сервисы, которые делают решение практичным для реальной эксплуатации:

- **Client-side canonicalization** для локальной подготовки и прямой отправки
- **Ingress API** как опциональный сервис стандартизации и подготовки
- **Finality Watcher** для отслеживания необратимости
- **Receipt Service** для выдачи подтвержденного результата
- **Audit API** для поиска и проверки истории

## 6. How it works

### Step 1. Prepare

Клиентское приложение или backend может работать в двух режимах.  
Либо данные канонизируются и хэшируются локально, после чего request отправляется напрямую в блокчейн. Либо те же шаги выполняет Ingress API как вспомогательный сервис подготовки.

### Step 2. Anchor

Система публикует подтверждение:

- либо как single commitment
- либо как batch root для большого массива записей

### Step 3. Track finality

После включения транзакции платформа отслеживает irreversible finality, чтобы отделить сам факт включения от окончательного подтверждения.

### Step 4. Issue receipt

Когда запись достигает finality, клиент получает receipt с ключевыми метаданными подтверждения.

### Step 5. Audit and verify

Через Audit API можно найти запись, восстановить цепочку подтверждения, проверить статус и экспортировать результат для внешней проверки.

## 9. Platform components

### On-chain anchoring layer

Фиксирует commitments и batches, связывает записи со schema/policy-контекстом и поддерживает прозрачный lifecycle.

### Ingress API

Готовит входящие данные к подтверждению, нормализует формат и выдает подготовленный action payload. Используется как удобный сервис подготовки, но не является обязательной точкой входа.

### Client SDK / Local preparation layer

Позволяет канонизировать данные, формировать hashes и собирать action payload прямо на стороне клиента или backend без обязательного вызова Ingress API.

### Finality Watcher

Следит за состоянием подтверждения, отслеживает inclusion и finality, помогает отделить промежуточное состояние от окончательного.

### Receipt Service

Возвращает проверяемый receipt после достижения finality.

### Audit API

Позволяет искать записи по request id, external reference, transaction id, batch id и другим идентификаторам, а также получать proof chain.

### Архитектура DFS-хранения

DFS в deNotary построен как отдельный слой хранения с четким разделением control plane и data plane.

- **Metadata API** управляет профилями, квотами, подтверждением оплаты, метаданными, recovery-данными, логикой размещения и download-plan
- **Storage nodes** хранят зашифрованные chunks, отдают их только по авторизованным токенам и регулярно подтверждают свое состояние через heartbeats и replica reports
- **Client-side encryption** остается обязательным принципом: plaintext не покидает клиент, backend не расшифровывает содержимое
- **Cluster topology** рассчитана на многонодовое развертывание, где metadata layer и storage nodes работают как отказоустойчивый распределенный контур

Это позволяет связать проверяемую фиксацию данных в deNotary с защищенным внешним хранением, восстановлением и recovery workflow без отказа от криптографической модели доверия.

## 16. FAQ

### Что именно подтверждает deNotary?

deNotary подтверждает факт существования и неизменности цифровой записи в определенный момент времени, а также хранит проверяемую историю подтверждения через API и receipts.

### Подходит ли это только для документов?

Нет. Платформа подходит для документов, логов, сертификатов, событий, AI-output, dataset snapshots и других цифровых артефактов.

### Чем отличается single от batch?

Single используется для индивидуальной записи.  
Batch нужен для массовых потоков, где много записей объединяются в merkle root, чтобы сохранить проверяемость и повысить эффективность.

### Можно ли интегрировать deNotary в существующий продукт?

Да. Платформа может быть встроена в уже работающие сервисы и процессы как через API, так и через локальную клиентскую логику подготовки и прямой blockchain submit.

### Что получает клиент после подтверждения?

Клиент получает request metadata, finality-aware receipt и audit path, который можно использовать для проверки, экспорта и последующего анализа.
